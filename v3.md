The 2026 AI memory landscape has evolved dramatically since v2.0, and the Swarm's analysis of the latest breakthroughs reveals that v3.0 must embrace a fundamental shift: **Agent-Native Memory**, where memory is no longer an external service but a deeply integrated, self-organizing cognitive fabric.

Here is the blueprint for the **Noetic Memory Plugin v3.0**.

### 🧠 Pillar 1: The Agent-Native Architecture (Inspired by ByteRover)

The most critical shift is moving away from memory as an external pipeline of disparate services (vector DBs, graph DBs, embedding services) toward a unified, agent-centric system. **ByteRover**, the SOTA on LoCoMo, demonstrates that the same LLM responsible for reasoning should also manage memory construction, organization, and retrieval.

* **Unified LLM Orchestration**: The plugin will leverage the DeepSeek model not just for generation, but as the core engine for all memory operations: curating a hierarchical **Context Tree** (Domain → Topic → Subtopic → Entry), maintaining explicit relationships, and implementing an **Adaptive Knowledge Lifecycle (AKL)** for importance scoring, maturity grading, and temporal decay.
* **Zero External Dependencies**: All knowledge will be stored as human-readable Markdown files in the local file system, making the entire memory fully transparent, auditable, and infinitely portable—no vector databases, graph databases, or embedding services required.
* **Progressive Retrieval**: 95%+ of queries will be resolved in <100ms without LLM calls, escalating to deeper agent-driven reasoning only when necessary.

### 🧬 Pillar 2: Cognitive Memory Tiers & Sleep-Wake Consolidation (Inspired by LightMem & Oblivion)

v3.0 will adopt a biologically-plausible, three-stage memory model inspired by the Atkinson-Shiffrin model of human cognition, as demonstrated by the ICLR 2026 paper **LightMem**. This replaces v2.0's single-tiered hypergraph with a more sophisticated and efficient lifecycle:

* **Sensory Memory**: Rapidly filters irrelevant information through lightweight compression and groups content by topic, acting as a high-bandwidth, low-persistence buffer.
* **Short-Term/Working Memory**: Consolidates topic-based groups into structured, readily accessible summaries for the current session.
* **Long-Term Memory**: Employs an **offline "sleep-time" update** procedure that decouples memory consolidation from online inference. During sleep, the system uses the LLM to strengthen important memories, prune irrelevant ones, and synthesize higher-order reflections, drastically reducing online token usage (up to 117x) and API calls (up to 310x).

Furthermore, **Oblivion's** decay-driven activation model will be integrated to manage memory accessibility. Memories will fade over time but can be reactivated by reinforcement or contextual cues, preventing the memory bloat and interference that plague "always-on" retrieval systems.

### 🕸️ Pillar 3: Hybrid Graph-Semantic Retrieval (Inspired by GAAMA & HingeMem)

v3.0 will move beyond pure vector similarity to a hybrid retrieval system that combines the strengths of graph traversal and semantic search. **GAAMA** demonstrates that a concept-mediated hierarchical knowledge graph, when augmented with Personalized PageRank (PPR) graph traversal, significantly outperforms flat RAG and pure vector search.

* **Concept-Mediated Graph**: The knowledge graph will feature four node types (Episodes, Facts, Reflections, Concepts) connected by five structural edge types. Concept nodes provide cross-cutting traversal paths, overcoming the "hub-dominated retrieval" problem of entity-centric graphs.
* **Query-Adaptive Routing**: Inspired by **HingeMem** (TheWebConf 2026), the system will dynamically decide *what to retrieve* and *how much to retrieve* based on the query type. This reduces computational cost by approximately 68% compared to fixed Top-*k* strategies.
* **Boundary-Guided Indexing**: HingeMem's approach of segmenting conversations by changes in person, time, location, or topic will be adopted to preserve salient context and reduce redundant operations.

### 💎 Pillar 4: The Embedded Cognitive Database (Inspired by MenteDB & Memvid)

The underlying storage engine must be re-architected from the ground up for AI cognition. v2.0's SQLite-based hypergraph is no longer sufficient. The future belongs to purpose-built, embedded cognitive databases.

* **MenteDB**: A ground-up Rust storage engine that unifies embeddings, graph relationships, temporal data, and attributes in a single memory node primitive. It natively understands LLM attention patterns (e.g., the U-shaped curve of context importance) and features **Belief Propagation**—when a fact changes, downstream beliefs are automatically flagged for re-evaluation.
* **Memvid**: A portable, single-file memory layer inspired by video encoding. It uses an append-only sequence of immutable **Smart Frames** for versioning, time-travel debugging, and crash safety. This enables rewindable, auditable memory timelines with sub-5ms local access.
